## Summary
The branch expanded the Anki MCP server from a small set of tools to comprehensive AnkiConnect coverage (51 tools total), migrating all Anki operations from bash scripts to named MCP tools and removing the anki.py script entirely. A permission model was established where read operations are silent and destructive operations prompt the user, with specific whitelist entries for trusted skill invocations like update_note_fields in the red-edit skill. Logging was unified across the launcher and server via a shared make_logger utility, and MCP-level call logging was added by wrapping tool registrations in core.py to enable usage statistics without inflated counts. The architecture shifted to model-agnostic card generation: the system now queries the live model registry and auto-selects or creates models based on card content, hiding cloze handling and model management behind the MCP layer. Two high-level skills (red-edit and purple-delete) were simplified by replacing hardcoded field names and model-specific branching with generic field scanning via a single get_flagged_notes MCP call, letting the MCP layer own the technical details. New model-customization tools (update_model_templates, update_model_styling, store_media_file with URL support) were added to support rich card design workflows. A local resource system was introduced via FastMCP's resource protocol: a tools/ref/ directory holds curated reference documentation (template-reference.md, add-notes.md) registered in resources.py, eliminating external doc fetches and token waste while keeping reference material co-located with the tools that need it. The overarching architectural direction is to build rich, opinionated atomic building blocks in the MCP server so that skills and deck files can operate at a higher level of abstraction with less technical boilerplate.
